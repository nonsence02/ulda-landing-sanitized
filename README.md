# ULDA Landing Public Mirror

Публічна санітизована копія академічного лендингу бакалаврської роботи про безстанову автентифікацію, мінімальне розкриття метаданих та протоколи анонімної взаємодії клієнтів у вебзастосунках.

## Призначення репозиторію

Цей репозиторій створено як безпечну публічну версію навчального проєкту. У ньому залишено лише фронтенд-частину лендингу, придатну для демонстрації структури Next.js-проєкту, етапів підготовки академічної сторінки та подальшого розгортання на GitHub Pages.

## Що було вилучено під час санітизації

- історію приватного репозиторію та службову папку `.git`;
- файли середовища та потенційно чутливі конфігурації;
- API-роути, доступ до бази даних і серверні інтеграції;
- логіку надсилання email, автентифікацію та інші залежності від секретів;
- артефакти збірки й зайві службові файли.

## Структура проєкту

- `app/` - маршрути App Router, metadata, глобальні стилі та головна сторінка;
- `components/` - повторно використовувані React-компоненти інтерфейсу;
- `components/ui/` - базові UI-компоненти;
- `lib/` - конфігурація сайту, утиліти й структурований контент сторінки;
- `docs/` - інструкції з документування, архітектурні нотатки та workflow docs generation;
- `public/` - статичні ресурси;
- `.github/workflows/` - GitHub Actions для деплою та перевірки документації;
- `typedoc.json` - конфігурація автоматичної генерації API-документації;
- `.env.example` - приклад файлу середовища без секретів;
- `.gitignore` - правила виключення службових і чутливих файлів із Git;
- `Dockerfile` - контейнеризація застосунку;
- `LICENSE` - ліцензія на код;
- `package.json` - залежності та скрипти запуску.

## Локальний запуск

```bash
npm install
npm run dev
```

Застосунок буде доступний за адресою `http://localhost:3000`.

## Перевірка збірки

```bash
npm run build
```

Після збірки Next.js формує статичний експорт у папку `out/`, придатний для GitHub Pages та інших static hosting платформ.

## Документація коду

У репозиторії використовується `TSDoc` для публічних TypeScript-експортів і `TypeDoc` для автоматичної генерації HTML-документації.

Основні команди:

```bash
npm run docs:build
npm run docs:check
```

Згенерована документація потрапляє до `typedoc-site/`. Детальніше:

- `docs/generate_docs.md`
- `docs/architecture.md`
- `docs/component-interactions.md`

## Правила внеску щодо документації

- документуй усі нові публічні експорти через TSDoc-коментарі;
- для нетривіальної логіки додавай короткий `@remarks` або пояснювальний блок у відповідному модулі;
- якщо змінюється структура взаємодії компонентів або архітектурні рішення, оновлюй файли в `docs/`;
- перед створенням PR запускай `npm run docs:build`, щоб перевірити, що документація генерується без помилок.

## Розгортання на GitHub Pages

Проєкт налаштований на безпечний статичний деплой через GitHub Actions.

1. Запуш зміни в `main`.
2. У GitHub відкрий `Settings -> Pages`.
3. У секції `Build and deployment` вибери `Source: GitHub Actions`.
4. Дочекайся виконання workflow `Deploy to GitHub Pages`.

Після успішного workflow сайт буде доступний за адресою:

```text
https://nonsence02.github.io/ulda-landing-sanitized/
```

## Як це працює

- `next.config.mjs` використовує `output: "export"` для статичного експорту;
- під час GitHub Actions автоматично вмикаються `basePath` і `assetPrefix` для розміщення сайту в підшляху репозиторію;
- workflow `.github/workflows/deploy-pages.yml` збирає проєкт, генерує TypeDoc, копіює docs у статичний export і публікує все в GitHub Pages;
- локальна розробка не змінюється: `npm run dev` і далі працює без `basePath`.
