# Architecture Overview

## Призначення системи

Репозиторій містить публічний санітизований лендинг бакалаврської роботи. Це статичний Next.js App Router-проєкт без бекенду, секретів, автентифікації чи інтеграцій із базою даних.

## Основні архітектурні рішення

### 1. Статичний export замість серверного рантайму

Проєкт розгортається через GitHub Pages, тому використовує статичний export. Це спрощує публікацію, усуває потребу в сервері та робить репозиторій безпечним для відкритого доступу.

### 2. Єдине джерело контенту

Білінгвальний контент зосереджено в `lib/thesis-content.ts`. Такий підхід дає змогу:

- централізовано оновлювати тексти;
- не дублювати контент по компонентах;
- перемикати мову без окремого бекенду або CMS.

### 3. Тонкі UI-компоненти

Базові елементи інтерфейсу винесено в `components/ui/`. Вони відповідають за стилізацію та повторне використання, але не містять бізнес-логіки.

### 4. Окремий конфіг сайту

SEO, зовнішні посилання, назви роботи та службові метадані зберігаються в `lib/site.ts`. Це спрощує підтримку metadata, robots, sitemap і Pages deployment.

## Нетривіальна логіка

### Перемикач мов

`app/page.tsx` рендерить сторінку на основі локального стану мови. Це навмисно зроблено на клієнті, щоб отримати легке двомовне перемикання без маршрутизації або дублювання сторінок.

### Metadata-маршрути

`app/robots.ts` і `app/sitemap.ts` позначені як `force-static`, оскільки проєкт збирається у static export режимі. Це необхідно для сумісності з GitHub Pages.

### Документація як частина quality gate

Генерація документації через `TypeDoc` винесена в окрему команду та CI workflow. Якщо public API змінюється так, що docs більше не генеруються, це вважається помилкою збірки документації.
